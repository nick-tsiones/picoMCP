## 1. Bugs

### B-1: CLI entry point is `.mjs`, not `.js` (10/10)

**Severity:** High | **Fix complexity:** Low

`node packages/picoMCP/dist/index.js` fails with `MODULE_NOT_FOUND` because the build outputs `dist/index.mjs`. Every subagent tripped over this on first use.

**Fix:** Add a `"bin"` field to `packages/picoMCP/package.json`: `"picomcp": "./dist/index.mjs"`. Also add a small CJS shim at `dist/index.js` that dynamic-imports the ESM entry, and document the correct invocation path in README and AGENTS.md.

### B-2: Project boundary rejects git worktree paths (9/10)

**Severity:** High | **Fix complexity:** Medium

`resolveProjectRoot()` walks up from CWD looking for `.qd`. When working in a git worktree outside the original project directory, the algorithm finds the original repo's `.qd` and then rejects worktree paths as "outside the project boundary." Every command requires `--root /tmp/picoMCP_worktrees/<game>`.

**Fix:**

- Support a `PICOMCP_ROOT` environment variable as fallback
- Improve `resolveProjectRoot` to also search for `.git` files (worktrees have `.git` as a file, not a directory) and prefer the closest match
- When the error fires, include the detected project root and suggest using `--root <dir>`
- Default to the parent directory of the target `.p8` file when no `.qd` is found and `--root` is not specified

### B-3: Cross-device rename fails on save (1/10 -- Asteroids)

**Severity:** Medium | **Fix complexity:** Low

When `/tmp` and the target directory are on different filesystems, `fs.rename` of the temp file fails with `EXDEV: cross-device link not permitted`.

**Fix:** In `cart_repo.ts:274`, add a fallback path: if `fs.rename` fails with `EXDEV`, use `fs.copyFile` + `fs.unlink`. Alternatively, write temp files into the same directory as the target file (`path.dirname(filePath)`).

### B-4: Lint false positives on table constructor fields (2/10 -- Asteroids, Galaga)

**Severity:** Medium | **Fix complexity:** Medium

The linter flags `"x" is assigned without "local"` for code like `local a = { x = rnd(128), y = rnd(128) }`. Table constructor key-value pairs are field definitions, not variable assignments.

**Fix:** Update the static analysis parser to track whether the current token position is inside a table constructor `{...}` and suppress the non-local assignment warning for constructor keys.

### B-5: Headless run times out silently, loses trace data (4/10)

**Severity:** Medium | **Fix complexity:** Medium

The default 10-second timeout (`DEFAULT_TIMEOUT_MS = 10000`) is too short for moderate frame counts (e.g., 360 frames at 30fps = 12s). When a run times out: `success: false` even though all requested frames completed; `performance` is null, `traces` is empty, `logPath` is null; screenshot may be captured but result reports failure.

**Fix:**

- Expose `--timeout-ms` as a CLI/MCP option
- Flush the log file periodically (every N frames) rather than only at end
- When capture-at frame was reached before timeout, report `captureSuccessful: true`
- Distinguish "timeout" from "runtime error" in result output

### B-6: `hasSprites`/`hasMap`/`hasMusic` inaccurate for runtime-generated assets (4/10)

**Severity:** Low | **Fix complexity:** Low

- `hasSprites: false` when sprites are created via `sset()` at runtime
- `hasMap: false` when map is populated via `mset()` in `_init()`
- `hasMusic: true` when the `__music__` section has only all-zero data

**Fix:** The overview is technically correct (reflects on-disk state) but misleading. Add a note to `read` output distinguishing on-disk vs runtime data. Fix `hasMusic` to check `music.some()` for non-zero data.

### B-7: No music CLI commands (1/10 -- Tetris)

**Severity:** Medium | **Fix complexity:** Medium

The CLI has full `sfx` support (get/set/list) but zero `music` commands. Music is a first-class PICO-8 asset and needs parity.

**Fix:** Add `dispatchMusic` to `dispatch-subcommands.ts` with `music get`, `music set`, `music list` subcommands, mirroring the SFX implementation.

### B-8: Index convention confusion: 0-based CLI vs 1-based MCP (4/10)

**Severity:** Low | **Fix complexity:** Documentation

CLI uses 0-based indices for sprites/SFX/flags, but map coordinates are 1-based (user-facing). MCP tools use 1-based everywhere. Users mix them up.

**Fix:** Document the convention per-command in `--help` output. Consider standardizing on 0-based everywhere and making MCP translation explicit.

## 2. Missing Commands

### MC-1: `picomcp init` / `create` / `new` -- Cartridge scaffolding (10/10) [P0]

Every subagent had to manually construct the `.p8` header, section markers, and boilerplate code. There is no command to create a fresh cartridge.

```bash
picomcp init mygame.p8                    # minimal valid cartridge
picomcp init mygame.p8 --template shooter # genre-specific boilerplate
```

Should create a `.p8` with `version 42`, all section headers (`__lua__`, `__gfx__`, `__gff__`, `__map__`, `__sfx__`, `__music__`), and `_init()`, `_update()`, `_draw()` stubs.

### MC-2: `--code-file` flag on write/edit (10/10) [P0]

Every subagent struggled with shell escaping multi-line Lua code. The current workaround (`--code "$(cat game.lua)"`) is fragile with special characters like `$`, backticks, and backslashes.

```bash
picomcp write cart.p8 --code-file game.lua
picomcp edit cart.p8 append --code-file overlay.lua
```

Also support `--code=@file.lua` syntax (like curl).

### MC-3: `picomcp dev` / `watch` -- Watch mode with auto-reload (10/10) [P0]

The edit -> write -> parse -> lint -> run cycle is 4+ CLI invocations per iteration. Every subagent called this the single biggest productivity blocker.

```bash
picomcp dev cart.p8                     # watch file, auto-parse/lint/run
picomcp dev cart.p8 --frames 120        # headless run each time
picomcp dev cart.p8 --capture screen    # capture screenshot per iteration
```

Should watch the `.p8` file (and optionally linked source files), re-run parse+lint on save, and optionally execute a headless run with results reported inline.

### MC-4: `picomcp check` -- Combined parse+lint+size (9/10) [P1]

Currently requires three separate commands for basic validation. A single `check` command would cut 3 CLI calls to 1.

```bash
picomcp check cart.p8   # parse + lint + size in one unified report
```

Additionally, `picomcp validate cart.p8` could run a single-frame headless execution to catch PICO-8-specific syntax/API errors that the linter misses.

### MC-5: Batch / manifest cartridge creation (8/10) [P1]

A complete game required 15-30 separate CLI calls (write code, set 10+ sprites, 4+ SFX, map data). A single manifest-based creation would reduce this to 1 call.

```bash
picomcp build recipe.json --output game.p8
```

Where `recipe.json` contains code (with `@file.lua` references), sprites (with `fill` shortcuts), SFX (with compact notation), map data (with `@maze.txt` references), and flags.

### MC-6: Sprite preview / ASCII render (8/10) [P1]

No way to visually verify a sprite without opening PICO-8. `sprite get` returns 64 integers. `sprite export` writes a PNG requiring an image viewer.

```bash
picomcp sprite get cart.p8 --index 0 --preview  # ASCII art, ANSI terminal colors
picomcp sprite show cart.p8 --all               # full sprite sheet preview
```

Render with ANSI truecolor escape codes using the PICO-8 palette, or use block characters for a text-grid representation.

### MC-7: Sprite fill / solid-color (8/10) [P1]

Setting a solid-color 8x8 sprite requires typing 64 comma-separated values. For Tetris blocks and breakout bricks, this was the most tedious task.

```bash
picomcp sprite fill cart.p8 --index 0 --color 8
picomcp sprite fill-range cart.p8 --start 1 --end 7 --colors 8,9,10,11,12,13,14
```

### MC-8: SFX preview / compact notation (7/10) [P1]

No way to hear an SFX without running the full cartridge. The SFX JSON format is verbose.

```bash
picomcp sfx play cart.p8 --index 0                  # headless playback
picomcp sfx preview cart.p8 --index 0               # text description of notes
picomcp sfx tone cart.p8 --index 0 --notes "C4,E4,G4" --speed 8  # musical shorthand
picomcp sfx generate cart.p8 --index 1 --type pew    # common SFX presets
```

### MC-9: Map drawing tools (7/10) [P1]

Map editing is limited to individual cell get/set and JSON region arrays. Drawing a maze or racetrack requires external script generation.

```bash
picomcp map fill cart.p8 --x 1 --y 1 --w 16 --h 15 --tile 1
picomcp map from-text cart.p8 --x 1 --y 1 --legend "W=1 .=0" @maze.txt
picomcp map draw-line cart.p8 --x1 10 --y1 20 --x2 80 --y2 40 --tile 2 --width 8
picomcp map draw-rect cart.p8 --x 0 --y 0 --w 16 --h 12 --tile 3
```

### MC-10: Cartridge diff (7/10) [P2]

No way to see what changed between two versions of a cartridge. Especially important for MCP agent workflows.

```bash
picomcp diff before.p8 after.p8 --json
picomcp diff before.p8 after.p8 --deltas-only
```

Should show: code changes (unified diff), sprite pixel changes (which sprites, which pixels), map cell changes, SFX note changes.

### MC-11: Sprite copy / mirror (6/10) [P2]

Ghost animation frames, directional sprites, and mirrored versions require re-typing nearly identical pixel arrays.

```bash
picomcp sprite copy cart.p8 --from 5 --to 9
picomcp sprite mirror cart.p8 --index 5 --horizontal
picomcp sprite flip cart.p8 --index 5 --vertical
picomcp sprite recolor cart.p8 --index 5 --map '{"3":"12","6":"11"}'  # palette swap
```

### MC-12: `picomcp test` -- Automated game testing (5/10) [P2]

Headless runs only report exit codes and frame counts. No way to assert game state at specific frames.

```bash
picomcp test cart.p8 --assert "score > 0" --frames 300
picomcp test cart.p8 --assert "state != 'gameover'" --frames 120
picomcp test cart.p8 --golden-screenshot expected.png --capture-at 60
```

### MC-13: `edit insert` / `edit delete` line operations (3/10) [P2]

`edit range` replaces lines. `edit append` adds to end. No way to insert lines mid-file or delete lines without replacing.

```bash
picomcp edit insert cart.p8 --at 42 --code "new_function()"
picomcp edit delete cart.p8 --from 10 --to 15
```

### MC-14: `sprite draw` -- primitive shapes on sprites (3/10) [P2]

Drawing shapes directly on sprites via CLI.

```bash
picomcp sprite draw cart.p8 --index 5 --shape circle --cx 4 --cy 4 --radius 3 --color 7
picomcp sprite draw cart.p8 --index 5 --shape rect --x 1 --y 1 --w 6 --h 6 --color 12
```

### MC-15: SFX set-range (bulk SFX) (3/10) [P2]

`sprite set-range` exists but `sfx set-range` does not. Setting 4+ SFX requires 4+ separate CLI calls.

### MC-16: `picomcp mem` -- Token/memory profiling (5/10) [P2]

The current `tokenCount` uses a character-count heuristic (`code.replace(/\s+/g, "").trim().length`) that doesn't match PICO-8's actual token counting. PICO-8's real constraint is 8192 tokens, not 65536 characters.

**Fix:** Implement a proper PICO-8 Lua tokenizer that counts tokens the way PICO-8 does (keywords=1, identifiers=1, operators=1, string literals=1 each). Integrate with `picomcp size` or add `picomcp mem`.

## 3. Workflow Pain Points

### WP-1: `--root` required on every command (10/10) [Severity: High]

Working outside the main project directory (e.g., in git worktrees) requires `--root <path>` on every single CLI invocation. This adds ~40 characters of boilerplate to every command.

**Fix:** Support `PICOMCP_ROOT` environment variable. Add `.picomcprc` config file. Auto-detect the cartridge file's parent directory as the root when no `.qd` is found.

### WP-2: No compact SFX notation (8/10) [Severity: Medium]

The SFX JSON format requires fully expanded note objects. A compact shorthand (`"C4,E4,G4"`) and preset generators (`--type pew`, `--type explosion`) would make sound design much faster.

### WP-3: Sprite data format is opaque (10/10) [Severity: High]

The `--pixels "0,0,7,7,0,0,..."` format requires typing 64 numbers per sprite by hand. This is the single most cited friction point -- every subagent struggled with it. Some resorted to programmatic sprite generation in Lua (`sset()` calls in `_init()`) to avoid the CLI entirely.

**Fix (priority ordered):**

1. `sprite fill` / `sprite fill-range` for solid-color sprites
2. `sprite draw --ascii` accepting text-art (`.` = 0, `#` = 7, `0`-`f` hex)
3. `sprite import` already exists for PNGs but should accept individual 8x8 PNGs
4. Visual sprite editor (TUI or web-based)

### WP-4: No watch / live-reload mode (10/10) [Severity: High]

Iterative game development requires manually re-running parse/lint/run after every code change. The feedback loop is 5-15 seconds per change. A watch mode would collapse this to near-instant.

### WP-5: Multi-command orchestration requires shell scripting (10/10) [Severity: High]

Setting up a complete game required stringing together 15-30 CLI calls in bash scripts. Batch operations or a manifest would streamline this.

### WP-6: Parse doesn't catch PICO-8-specific errors (5/10) [Severity: Medium]

`parse` and `lint` report success even for code with PICO-8 syntax errors (e.g., inline `if` without `end`). The error is only caught by the actual headless run, wasting time.

**Fix:** Integrate a PICO-8-compatible Lua parser, or add a `--check` flag to `run` that only validates syntax without executing frames.

### WP-7: Runtime error messages not structured (6/10) [Severity: Medium]

When a headless run encounters a Lua error, the `RunCartResult` doesn't parse PICO-8's error output into structured fields (`line`, `tab`, `message`). The error text is buried in stdout/stderr.

**Fix:** Parse PICO-8 error patterns (e.g., "runtime error line N tab M: <msg>") and surface them in a dedicated `errors[]` field in the JSON result.

## 5. Better Tooling Ideas

### 5.1 Visual Sprite Editor (TUI)

A terminal-based sprite editor that shows the 8x8 grid with PICO-8's 16-color palette, lets you draw pixels with keyboard (arrow keys + number keys for colors), and saves via the CLI. This would be the single most transformative feature for asset creation.

### 5.2 Map Editor with Drawing Tools

A TUI or web-based map editor showing the 128x64 tile grid, with paintbrush, line, rectangle, fill, and stamping from the sprite sheet. Track design and maze creation currently require external scripts.

### 5.3 Project Templates

```
picomcp template list                           # show available templates
picomcp init mygame.p8 --template platformer    # scaffold genre-specific game
picomcp init mygame.p8 --template shmup         # with HUD, scrolling, enemies
```

Templates should include: suggested sprite layout, SFX placeholders, commented code patterns, common game-loop patterns.

### 5.4 Cartridge Packager / Bundler

Combine multiple `.lua` source files, sprites from PNGs, and map data into a single `.p8` cartridge. Would enable modular development: separate Lua files for game logic, separate sprite sheet PNG, separate map data. Single `picomcp build` assembles the final cartridge.

### 5.5 Debug Adapter Protocol (DAP) Support

For IDE integration: set breakpoints, step through Lua code, inspect variables. The headless runtime driver already wraps `_init/_update/_draw` -- extending it to support breakpoints and variable inspection would make it a full debugging tool.

### 5.6 Live Preview / Hot Reload

`picomcp dev` that watches the `.p8` file and hot-reloads PICO-8. When a code change occurs: re-write the cartridge, re-run headless (or launch PICO-8 graphically), show parse/lint/size results inline, and optionally capture a screenshot/GIF. This would cut iteration time from 5-15 seconds to <1 second.

### 5.7 PICO-8 API Interactive Reference

`picomcp ref api --search "spr"` or `picomcp ref api --function "spr"` to pull up specific function docs. The current `ref api` dumps all 126 functions as a massive JSON blob -- useful for MCP agents, unreadable for human developers.

### 5.8 Code Snippets / Library Imports

```
picomcp include cart.p8 --module collision     # import collision detection helpers
picomcp include cart.p8 --module particles     # import particle system boilerplate
```

A library of common PICO-8 utility functions (AABB collision, vector math, state machines, particle systems) importable via CLI.

### 5.9 Cartridge Meta / Packaging

```
picomcp meta set cart.p8 --title "Asteroids" --author "Name" --version "1.0"
picomcp package cart.p8                        # create distributable zip
```

Support for `__label__` section for cartridge labels, and packaging for distribution.

## 6. Edit -> Parse -> Lint -> Run Cycle Issues

### CYC-1: Four separate commands per iteration (10/10)

Each code change requires: `write` -> `parse` -> `lint` -> `run` (4 CLI invocations). Each with `--root`, cartridge path, and flags. ~10 seconds of overhead per iteration.

**Fix:** `picomcp check` (parse+lint+size in one) and `picomcp dev` (watch mode) are the priority solutions.

### CYC-2: No atomic write-and-validate (5/10)

`write` saves to disk regardless of parse errors. Invalid code can be persisted, and the user doesn't discover the error until the next `parse` call.

**Fix:** `write` and `edit` commands should auto-parse and auto-lint after mutation by default, reporting results inline. Add `--skip-validate` to suppress.

### CYC-3: Parse doesn't catch PICO-8-specific errors (5/10)

The in-browser parse checks balanced brackets/quotes but doesn't validate against PICO-8's actual Lua parser. PICO-8-specific syntax (e.g., inline `if` without `end`, `spr(999)` out-of-range) passes parse but fails at runtime.

**Fix:** Add a `picomcp compile` command that uses PICO-8's parser in headless mode to check syntax without running frames. Or integrate a PICO-8-compatible Lua parser.

### CYC-4: Lint is basic, misses PICO-8 pitfalls (6/10)

`lint` doesn't catch: PICO-8-specific operator mistakes (`!=` instead of `~=`), API misuse (wrong arg count to `spr()`), missing lifecycle functions, token limit approaches, unused sprites/SFX, unreachable code, or global variable conflicts with PICO-8 builtins.

**Fix:** Add PICO-8-specific lint rules. Allow configurable rule sets via `.picomcp-lintrc` or `--severity` flag. Add `--verbose` to list checked rules.

### CYC-5: No incremental validation (4/10)

After an `edit replace`, the user must manually run `parse` and `lint` as separate commands. If the replacement introduced a syntax error, it's not discovered until later.

**Fix:** Every `edit` command should auto-parse and auto-lint after mutation. Or add a `--validate` flag to write/edit.
